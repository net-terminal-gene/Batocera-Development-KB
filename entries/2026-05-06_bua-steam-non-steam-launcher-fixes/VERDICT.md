## Status: IMPLEMENTED (branch ready, pending test + PR)

## Summary

Non-Steam game launchers generated by `create-steam-launchers2.sh` had no sound and broken controller input because they invoked Proton directly, bypassing the Steam Linux Runtime container. The fix uses the same launch chain Steam itself uses: `SteamLinuxRuntime_4/_v2-entry-point --verb=run -- Proton - Experimental/proton run`. Validated on-device with 5 games (eXceed 2nd, Maldita Castilla, SenXin Aleste, eXceed 3rd, eXceed Gun Bullet Children) - sound, controllers, and artwork all working correctly.

Both fixes (SLR launch chain + artwork dimension fallback) are implemented in the local repo on branch `fix/steam-non-steam-launcher-slr`.

## Plan vs reality

- Fix 1 required additional env vars not anticipated in initial research:
  - `PROTON_NO_STEAM_OVERLAY=1` alone was insufficient; needed `WINEDLLOVERRIDES="lsteamclient=d;steam.exe=d"` to suppress the Wine steamclient assertion entirely.
  - Additional identity vars needed: `SteamAppId`, `SteamGameId` (alongside existing `STEAM_COMPAT_APP_ID`)
  - The discovery came from reading Steam's own `compat_log.txt` which showed the SLR launch prefix.
- `detect_proton()` function was removed entirely - Proton Experimental is hardcoded as the default. This avoids issues where versioned Proton installs may not have full SLR compatibility.
- `SDL_GAMECONTROLLERCONFIG` hack was removed - SLR provides proper SDL environment natively.
- The correct upstream file is `steam/extra/create-steam-launchers2.sh` (deployed by `steam2.sh`), not the original `create-steam-launchers.sh`.

## Root Causes

1. Launcher template calls `"$PROTON_PATH" run "$EXE"` directly, missing the SteamLinuxRuntime container that provides audio and SDL environment
2. `sgdb_get_image()` hardcodes `dimensions=460x215` with no fallback

7. Added `WINEDLLOVERRIDES` to disable lsteamclient/steam.exe DLLs ā€” fully working
8. Applied to all 5 non-Steam games via template update ā€” universal fix confirmed
