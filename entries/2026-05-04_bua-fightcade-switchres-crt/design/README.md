# Design — Fightcade Switchres CRT Integration

## Architecture

### Game Launch Flow (Current)

```
fc2-electron (Fightcade UI)
  └─> xdg-open fcade://play/fbneo/sfiii3nr1
       └─> xdg-open shim (PR #143)
            └─> fcade.sh
                 └─> fcade binary (closed-source ELF)
                      └─> Resources/wine.sh
                           └─> wine fcadefbneo.exe <rom> (Wine + DX9)
```

### Game Launch Flow (With Switchres Integration)

```
fc2-electron (Fightcade UI)
  ├─ User in room → TEST GAME or ONLINE MATCH
  └─> xdg-open fcade://play/fbneo/sfiii3nr1
       └─> xdg-open shim (MODIFIED)
            ├─ 1. Parse URL → emulator=fbneo, rom=sfiii3nr1
            ├─ 2. /usr/bin/mame/mame -listxml sfiii3nr1 → 384x224@59.60
            ├─ 3. Patch FBNeo config for borderless fullscreen
            ├─ 4. switchres W H R -s -k  (switch CRT, keep on exit)
            ├─ 5. fcade.sh "$URL"  (returns immediately — fcade backgrounds)
            ├─ 6. Poll: while pgrep wine/emulator; sleep 2; done
            ├─ 7. Restore FBNeo config
            └─ 8. Restore resolution (xrandr back to ES mode)
                   └─ User returns to Fightcade room
                      (can launch another game → repeats from step 1)
```

**Critical: `fcade.sh` backgrounds the `fcade` binary** (`${FPY} ${PARAM} 2>&1 &`),
so it returns immediately. Cannot use `switchres -s -l "fcade.sh ..."` because
switchres would restore the resolution instantly. Must use `-s -k` (keep) and
handle restore via process polling.

### Fightcade Session Lifecycle

```
Fightcade.sh (port launcher)
  └─> fc2-electron starts (Fightcade client UI)
       ├─ User browses rooms, joins one
       ├─ ROM auto-downloads if missing
       ├─ TEST GAME ──> xdg-open fcade://play/fbneo/<rom>
       │    └─ Game runs until Game > Exit (manual)
       │    └─ Returns to Fightcade room
       ├─ ONLINE MATCH ──> xdg-open fcade://play/fbneo/<rom>
       │    └─ Game runs until rounds complete (automatic exit)
       │    └─ Returns to Fightcade room
       ├─ (can play more games, each triggers new fcade:// URL)
       └─ User closes Fightcade → sym_wine.sh cleans up
```

Both TEST GAME and ONLINE MATCH use the same `fcade://play/` URL dispatch.
The xdg-open shim fires for every game launch within a session. Resolution
switching is per-game (switch on launch, restore on exit, repeat).

### Injection Point

The xdg-open shim in `Fightcade.sh` (generated by `fightcade.sh` installer) is the single choke point for all game launches. Currently:

```bash
case "$1" in
    fcade://*)
        /path/to/fcade.sh "$1"
        ;;
esac
```

Modified to:

```bash
case "$1" in
    fcade://*)
        switchres_fightcade_wrap "$1"
        ;;
esac
```

### Process Monitoring Strategy

Because `fcade.sh` backgrounds `fcade` (which then spawns Wine + emulator), the
wrapper cannot simply wait for `fcade.sh` to exit. Strategy:

```bash
switchres_fightcade_wrap() {
    local URL="$1"
    # ... parse URL, lookup resolution, patch config ...
    
    switchres $W $H $R -s -k   # switch and keep
    
    # Launch (returns immediately because fcade backgrounds)
    $ADDONS_DIR/fightcade/Fightcade/emulator/fcade.sh "$URL"
    
    # Wait for the actual emulator process to appear, then monitor
    sleep 2  # let spawn chain settle
    while pgrep -f "fcadefbneo.exe|ggpofba-ng.exe|fcadesnes9x.exe|flycast.elf" >/dev/null 2>&1; do
        sleep 2
    done
    
    # Emulator exited (Game > Exit or match ended)
    restore_fbneo_config
    restore_display  # xrandr back to ES resolution
}
```

**Handles both TEST GAME and ONLINE MATCH identically:**
- TEST GAME: user manually exits via Game > Exit, pgrep sees process gone
- ONLINE MATCH: match ends, emulator exits on its own, pgrep sees process gone

**Edge cases:**
- Multiple rapid game launches: each xdg-open call is serial (shim blocks until emulator exits)
- Fightcade UI stays running behind switched resolution (acceptable, it's hidden on CRT)
- If Wine crashes, pgrep loop still catches exit

### Per-Emulator Matrix

| Emulator | Binary | Type | Resolution Source | Notes |
|----------|--------|------|-------------------|-------|
| fbneo | fcadefbneo.exe | Wine | MAME -listxml | DX9 Alt blitter, needs bVidDX9WinFullscreen=1 |
| ggpofba | ggpofba-ng.exe | Wine | MAME -listxml | Same pattern as fbneo |
| snes9x | fcadesnes9x.exe | Wine | Fixed 256x224@60.10 | SNES is always the same |
| flycast | flycast.elf | Native Linux | Fixed 640x480@59.94 or MAME | Dreamcast, native ELF |

### Resolution Lookup

Primary: `/usr/bin/mame/mame -listxml <rom>` (29ms, has exact refresh rate)
Fallback by hardware class:

| Hardware | Resolution | Refresh |
|----------|-----------|---------|
| CPS1/CPS2/CPS3 | 384x224 | 59.64 |
| Neo Geo MVS | 320x224 | 59.19 |
| SNES | 256x224 | 60.10 |
| Dreamcast | 640x480 | 59.94 |

### FBNeo Config Patching

Settings changed before launch, restored after:

| Setting | CRT Value | Purpose |
|---------|-----------|---------|
| nVidHorWidth | match ROM | Fullscreen width |
| nVidHorHeight | match ROM | Fullscreen height |
| bVidAutoSwitchFull | 1 | Auto-fullscreen on load |
| bVidDX9WinFullscreen | 1 | Borderless windowed (no Wine mode change) |
| bVidArcaderesHor | 0 | Don't let FBNeo try its own mode switch |
| nVidScrnAspectX | 4 | CRT aspect |
| nVidScrnAspectY | 3 | CRT aspect |
| bVidFullStretch | 1 | Fill the screen |
| bVidCorrectAspect | 0 | We handle aspect via switchres |

## Deployment Strategy

### Code Ownership

Mikey does not own `batocera-unofficial-addons` (BUA) or `Batocera-CRT-Script`.
The implementation must account for possible rejection from either repo.

### Priority Order

| Priority | Target Repo | Delivery | Condition |
|----------|-------------|----------|-----------|
| 1 (PoC) | batocera-unofficial-addons | PR to modify `fightcade/fightcade.sh` xdg-open shim | Testing now; PR if PoC validates fully |
| 2 | ZFEbHVUE/Batocera-CRT-Script | `extras/fightcade-crt-patch.sh` + curl one-liner | If BUA rejects; CRT-specific logic in CRT repo |
| 3 | Mikey's own repo (new or existing) | Standalone curl install script | If both upstream repos reject |

### Option A: BUA PR (current PoC path)

Modify `fightcade/fightcade.sh` to generate a switchres-aware xdg-open shim.
Graceful fallback: if `switchres` not present, shim behaves identically to today.

**Pros:** Single install step, users get it automatically
**Cons:** Requires BUA maintainer buy-in; adds CRT-specific code to a general addon

### Option B: CRT-Script patch (curl post-install)

A standalone script in `Batocera-CRT-Script` that patches the Fightcade xdg-open
shim after Fightcade is already installed:

```bash
curl -L https://raw.githubusercontent.com/ZFEbHVUE/Batocera-CRT-Script/main/extras/fightcade-crt-patch.sh | bash
```

Script behavior:
1. Verify Fightcade installed (`/userdata/system/add-ons/fightcade/` exists)
2. Verify switchres available (`/usr/bin/switchres` exists)
3. Replace `${HOME}/bin/xdg-open` with switchres-aware version
4. Idempotent, re-runnable after Fightcade updates

**Pros:** CRT concerns stay in CRT repo; no upstream approval needed
**Cons:** Extra step after Fightcade install; must re-run if Fightcade reinstalls

### Option C: Mikey's own repo

Same script as Option B, just hosted independently. Last resort if both
upstreams reject.

### Auto-hook possibility (Option B enhancement)

CRT-Script installer could auto-detect Fightcade and patch:
```bash
# In CRT-Script installer, after main setup:
if [ -d /userdata/system/add-ons/fightcade ]; then
    echo "Fightcade detected — applying CRT patch..."
    # patch xdg-open shim
fi
```

This gives zero-touch integration for users who install CRT-Script after Fightcade.
